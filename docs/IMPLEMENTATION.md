# Implementation Guide

Baseline knowledge and constraints for implementing AgentWiki. Read together with [DESIGN.md](./DESIGN.md).

## Hard constraints

1. **Subscription-only, no API key.** All LLM execution MUST go through Claude Code — either an interactive session (plugin commands) or headless mode (`claude -p`). Never call the Anthropic API or the Claude Agent SDK directly.
2. **Agent SDK is off-limits by policy.** Anthropic does not allow claude.ai subscription auth for products built on the Agent SDK ("Anthropic does not allow third party developers to offer claude.ai login or rate limits for their products" — [Agent SDK quickstart](https://code.claude.com/docs/en/agent-sdk/quickstart.md)). This is why the CLI is a thin wrapper around the user's own `claude` CLI, not an SDK app.
3. **The CLI never handles credentials.** It relies on the locally installed, already-authenticated `claude` CLI. If `claude` is missing or not logged in, fail fast with a clear install/login message. In CI, the user supplies `CLAUDE_CODE_OAUTH_TOKEN` (generated by `claude setup-token`, works with Pro/Max) as a secret.
4. **Runtime**: Node >= 20, ESM, TypeScript. Keep the wrapper's runtime dependencies at zero or near-zero (`node:child_process`, `node:fs` first).
5. **All wiki-generation intelligence lives in the plugin** (`commands/`, `skills/`), never in TypeScript. The CLI only parses args, checks preconditions, spawns `claude`, and streams output. This keeps interactive and headless behavior identical.

## Components

| Path | Responsibility |
|------|----------------|
| `commands/init.md` | `/agentwiki:init` — format selection, delegate to skill, write metadata |
| `commands/update.md` | `/agentwiki:update` — read metadata, diff-scope, delegate to skill |
| `skills/wiki-generation/SKILL.md` | The core workflow: discovery discipline, page structure, both output formats, init/update modes |
| `src/cli.ts` | `agentwiki init|update` → spawn `claude -p "/agentwiki:<cmd> ..."` |
| `examples/agentwiki-update.yml` | GitHub Actions template |

## Output format specs

### `llm-wiki` (default)

```
wiki/
├── index.md          # entry point: purpose, page map, reading order
└── <topic>.md        # flat topic pages, cross-linked with [[topic]]
```

- Cross-references use `[[wiki-link]]` syntax (Obsidian-compatible).
- `index.md` must link every page; every page links back to related pages.
- Compatible with the Karpathy LLM Wiki pattern used by the `documentation:llm-wiki` skill.

### `openwiki`

```
openwiki/
├── quickstart.md     # entry point
├── architecture/
├── operations/
├── cli/              # sections as needed
└── INSTRUCTIONS.md
```

- Mirrors the layout produced by [OpenWiki](https://github.com/langchain-ai/openwiki) so existing repos can migrate seamlessly.
- Optionally maintain the `<!-- OPENWIKI:START/END -->` marker block in the repo's `CLAUDE.md`.

## Metadata file: `agentwiki.json`

Written at the wiki root after every successful run; drives incremental updates.

```json
{
  "version": 1,
  "format": "llm-wiki",
  "lastRunCommit": "<git HEAD sha at generation time>",
  "lastRunAt": "<ISO-8601>"
}
```

Update mode: `git diff --name-only <lastRunCommit>..HEAD` scopes which wiki pages need revision. If the file is missing, `update` falls back to full `init` behavior (same as OpenWiki's `--update` in CI).

## Generation discipline (ported from OpenWiki)

Encode these rules in `skills/wiki-generation/SKILL.md` — they come from `openwiki/src/agent/prompt.ts` and are the heart of output quality:

- **Targeted discovery only**: repo tree, package/config files, README-like files, entrypoints, routing, schema files, representative files per domain. Never exhaustive reads, never `**/*` globs from root; prefer `rg --files` with excludes (`.git`, `node_modules`, `dist`, generated wiki output).
- **Evidence-grounded**: every important claim must trace to an inspected file or git history. Never invent files, APIs, or behavior.
- **Focused first pass**: init produces the entry page plus the smallest set of section pages that explain the repo, then stops. Refinement happens in update runs.
- **Update runs are surgical**: revise only pages affected by the diff; do not regenerate the whole wiki.

## Headless invocation (CLI → Claude Code)

Sketch (verify flags against current Claude Code docs before implementing):

```sh
claude -p "/agentwiki:update" \
  --permission-mode acceptEdits \
  --output-format stream-json
```

**To verify during implementation** (use the `claude-code-guide` agent or official docs):

- [ ] How plugin commands are made available in headless mode (`--plugin-dir`-style flag, marketplace pre-install, or inlining the prompt with `--append-system-prompt`). If plugins can't load headlessly, fallback: the CLI embeds the same command prompt text directly in `-p`.
- [ ] Exact `--output-format stream-json` event schema for progress rendering.
- [ ] Minimal `--allowedTools` set for wiki generation (Read/Glob/Grep/Write/Edit/Bash(git ...)).

## Testing

- **Unit (vitest)**: CLI arg parsing, precondition checks (claude binary detection), `agentwiki.json` read/write/fallback logic.
- **E2E (manual first)**: run `/agentwiki:init` on a small real repo in both formats; verify link integrity (`[[link]]` targets exist, index covers all pages). Automate link-integrity checks later as a plain Node script — no LLM needed.

## Publishing

- **npm**: `agentwiki` package (`files` already includes `dist` + plugin dirs).
- **Plugin marketplace**: repo root is an installable plugin (`.claude-plugin/plugin.json`). To distribute via `roboco-io` marketplace, add this repo to the org's marketplace.json (see `roboco-io/tools/plugins` if it exists).
